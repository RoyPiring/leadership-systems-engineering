## The Mission: Managing a AAA Studio's Crisis Quarter

### Project overview and objectives

In this project, I built a program management command center designed to stabilize a struggling AAA game development portfolio and demonstrate the ability to coordinate five cross-functional teams through a critical delivery period. The objective was to provide Division Director Tanaka with a clear operational picture, identify the root causes affecting the Leviathan and Shiva teams, and create a recovery strategy grounded in measurable project data.

The effort focused on synchronizing dependencies, managing escalations, validating recovery options, and producing PMP-aligned artifacts that could support executive decision-making while protecting the Q3 release window.

## Building the War Room Command Center

### Environment setup goals

I configured the Shinra War Room environment using Claude-based expert personas, a dedicated GitHub repository, GitHub Projects, and Linear workspaces to simulate a large-scale program management office. This created a centralized operating model for tracking sprint execution, dependencies, risks, and executive reporting.

The environment was designed to mirror the coordination challenges found in enterprise software and game development programs, where multiple teams operate independently but remain tightly coupled through shared milestones and release commitments.

## Loading the Scenario and PMBOK 8 Framework

### Scenario generation goals

I generated the Shinra scenario data pack, including team structures, project artifacts, sprint information, and active blockers. Using the Maya Kisaragi mentor persona, I completed a focused PMBOK 8 review and applied those principles while populating GitHub Projects and Linear with operational data.

This created a realistic simulation environment where project governance, planning, risk management, and recovery execution could be practiced against evolving program conditions.

### PMBOK 8 performance domains and artifact integration

The program applied seven key performance domains: Governance, Scope, Schedule, Finance, Stakeholders, Resources, and Risk.

A critical lesson was understanding traceability between artifacts. The Program Charter established approved objectives and success criteria, while the Risk Register tracked threats against those commitments. This linkage made it possible to identify Odin's marketing-driven feature requests as scope expansion because they lacked authorization within the approved charter baseline.

![Image](https://learn.nextwork.org/refreshed_maroon_timid_jujube/uploads/9376ae0f-7391-4cea-9c0a-5e3706c9ddc7_7dhstgm4)

### Leviathan velocity decline analysis

Leviathan experienced a severe velocity collapse, dropping from 24 completed points to 8 and then 2 across three cycles. This represented a structural delivery failure rather than a productivity issue.

Analysis revealed two primary causes: the loss of the team lead during active sprint execution and an ArtForge vendor dependency operating at reduced capacity while delivering critical assets several weeks late. Together, these issues blocked the majority of planned work.

The findings demonstrated that recovery depended on leadership replacement and vendor escalation rather than increased team effort.

## Detecting Cross-Team Conflicts and Running Change Control

### Artifact review objectives

I reviewed sprint plans, architecture decision records, implementation plans, and project documentation across all five teams to identify conflicts, dependencies, and schedule risks.

The review process also included formal change control analysis and velocity-based impact assessments to support executive decision-making.

![Image](https://learn.nextwork.org/refreshed_maroon_timid_jujube/uploads/9376ae0f-7391-4cea-9c0a-5e3706c9ddc7_264ub6v6)

### Conflict identified and change control rationale

The primary conflict involved Odin's inclusion of three unapproved marketing features. These additions increased planned work beyond realistic team capacity and introduced direct risk to a fixed esports delivery deadline.

Through formal change control, the request was evaluated against approved scope, schedule constraints, and program objectives. The analysis showed that retaining the features increased delivery risk without contributing to chartered outcomes, making scope reduction the preferred corrective action.

![Image](https://learn.nextwork.org/refreshed_maroon_timid_jujube/uploads/9376ae0f-7391-4cea-9c0a-5e3706c9ddc7_puut2xc1)

## Authoring the Program Charter

### Charter authoring objectives

I developed Program Charter v1 and refined it through multiple review cycles with both Maya Kisaragi and Director Tanaka personas. Feedback focused on measurable outcomes, governance clarity, quality targets, and escalation criteria.

The final charter established a clear baseline for scope management, performance measurement, and executive oversight.

![Image](https://learn.nextwork.org/refreshed_maroon_timid_jujube/uploads/9376ae0f-7391-4cea-9c0a-5e3706c9ddc7_a0apmqk4)

### Measurable success criteria added after Maya's review

Several qualitative objectives were converted into measurable performance targets. These included crash-free session thresholds, player retention goals, platform-specific quality requirements, service reliability targets, and defined escalation limits.

Ambiguous success measures were replaced with operational metrics that could be objectively tracked throughout delivery. This raised governance by ensuring progress and quality could be evaluated using consistent data rather than subjective interpretation.

![Image](https://learn.nextwork.org/refreshed_maroon_timid_jujube/uploads/9376ae0f-7391-4cea-9c0a-5e3706c9ddc7_av9ylvoc)

## Building the Risk Register, Stakeholder Matrix, and RACI

### Artifact building objectives

I produced a Risk Register with quantitative Expected Monetary Value (EMV) analysis, a Stakeholder Engagement Assessment Matrix using the Current-Desired model, and a program-level RACI matrix.

Together, these artifacts established governance structures for decision-making, stakeholder engagement, ownership accountability, and risk prioritization.

![Image](https://learn.nextwork.org/refreshed_maroon_timid_jujube/uploads/9376ae0f-7391-4cea-9c0a-5e3706c9ddc7_wcaf30nr)

### EMV calculation and risk response strategy

A major delivery risk was quantified using EMV analysis. The probability and financial impact of missing a critical deadline were evaluated to determine expected exposure and justify mitigation efforts.

The selected response strategy focused on protecting approved scope, performing early validation activities, and enforcing decision gates. Quantifying the financial benefit of mitigation provided a defensible business case for executive action and resource allocation.

## Generating the Executive KPI Dashboard and Visual Artifacts

### Dashboard generation objectives

I created an interactive executive dashboard using Chart.js along with supporting visual artifacts, including a dependency topology map, risk heatmap, and stakeholder power-interest analysis.

These visualizations transformed raw project data into decision-support tools that could be consumed quickly by both operational and executive audiences.

### Critical SPI teams and Director implications

Schedule Performance Index analysis revealed Leviathan and Shiva as the only critical delivery risks. Bahamut and Ifrit remained on track, while Odin showed manageable schedule pressure.

This finding shifted the executive conversation from program-wide concern to targeted intervention. The evidence showed that overall recovery depended on resolving a small number of high-impact blockers rather than restructuring all five teams.

## Delivering the Director's Pitch and Earning the Verdict

### Pitch deck objectives

I developed a five-slide HTML presentation supported by Web Speech API narration to communicate program status, recovery actions, risk exposure, and executive decisions required to stabilize delivery.

The presentation was structured to move from current state assessment to actionable recommendations supported by measurable evidence.

![Image](https://learn.nextwork.org/refreshed_maroon_timid_jujube/uploads/9376ae0f-7391-4cea-9c0a-5e3706c9ddc7_zngtv41z)

### Pitch deck structure and narrative rationale

The presentation followed a logical decision-making sequence: current status, root cause analysis, recovery plan, risk mitigation, and executive requests.

This approach ensured recommendations were supported by data before requesting approvals. The structure emphasized accountability, quantified impacts, and clearly defined the actions needed from leadership.

![Image](https://learn.nextwork.org/refreshed_maroon_timid_jujube/uploads/9376ae0f-7391-4cea-9c0a-5e3706c9ddc7_bfqt2onj)

### Responding to the Director's challenge and final verdict

When challenged on the proposed scope reduction, I used historical velocity data to demonstrate that delivering the full backlog exceeded the team's proven capacity. The analysis showed that removing selected features reduced delivery requirements to an achievable level.

The recommendation was approved with conditions, including funding alignment, leadership approvals, and predefined contingency triggers. The outcome reinforced the importance of evidence-based decision-making during program recovery efforts.

## Automating the Daily War Room Status Report

![Image](https://learn.nextwork.org/refreshed_maroon_timid_jujube/uploads/9376ae0f-7391-4cea-9c0a-5e3706c9ddc7_xfklgdoa)

### Automation and PMBOK 8 process adaptation

I implemented automated status reporting through GitHub Actions to eliminate manual report preparation and raise reporting consistency.

This adaptation aligned with PMBOK principles by reducing administrative overhead while increasing reliability. Instead of spending time compiling updates, project leadership could focus on escalation management, stakeholder communication, and recovery execution.

Because reporting data was sourced directly from the project system, accuracy increased while long-term maintenance effort decreased.

## Reflections and Key Takeaways

### Tools and concepts mastered

This project strengthened my ability to use GitHub Projects, Linear, Claude personas, and GitHub Actions as an integrated program management platform. It also reinforced practical application of PMBOK principles, quantitative risk analysis, executive reporting, governance frameworks, and process automation.

A major takeaway was learning how structured project data can be transformed into actionable executive insight.

### Time and challenges

The project was completed in approximately two hours. The most challenging portion involved configuring GitHub Actions and validating workflow permissions, formatting, and execution logic.

Troubleshooting automation issues highlighted the importance of precision when working with YAML-based pipelines and reinforced how small configuration errors can impact operational reliability.

### Personal learning reflection

This project provided hands-on experience building a multi-team command center, creating governance artifacts, automating reporting workflows, and managing recovery planning under simulated crisis conditions.

The next area I want to explore is predictive project analytics, specifically how forecasting models can be used to anticipate delivery risk, sharpen resource allocation decisions, and support large-scale program planning across distributed teams.
